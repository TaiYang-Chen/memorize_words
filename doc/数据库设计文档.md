# 数据库设计文档

本文档基于当前仓库静态代码整理，目标是帮助审查本地数据库设计是否存在缺陷。范围包含当前 app 实际编译进来的 Room 数据库、MMKV/SharedPreferences 本地状态，以及仓库中保留但未纳入当前构建的 `data-word` 旧词库数据库实现。

## 1. 构建范围与持久化边界

### 1.1 当前构建有效范围

`settings.gradle.kts` 只 include 聚合模块 `:data`，没有 include `:data-word`、`:data-wordbook` 等独立 data 子模块。`data/build.gradle.kts` 通过 `sourceSets` 将以下源码目录聚合进 `:data`：

- `data-account`
- `data-feedback`
- `data-floating`
- `data-practice`
- `data-study`
- `data-sync`
- `data-wordbook`

因此，当前 app 实际使用的是 `data-wordbook` 中的词书/词库 Room 设计；`data-word` 目录存在代码和独立 `WordDatabase`，但未被 `:data` 聚合，也未被 `settings.gradle.kts` include。

### 1.2 数据库命名策略

所有 Room 数据库均通过 `DestructiveRoomDatabaseFactory` 创建，物理数据库名来自 `NewArchitectureDatabase.contextName(contextName)`：

| 业务域 | Room Database | 文件名 | 构建状态 |
| --- | --- | --- | --- |
| 词书、词库、学习进度镜像 | `WordBookDatabase` | `memorize_words_arch_v1_wordbook.db` | 当前有效 |
| 学习记录、收藏、打卡 | `StudyDatabase` | `memorize_words_arch_v1_study.db` | 当前有效 |
| 练习、考试缓存、练习报告 | `PracticeDatabase` | `memorize_words_arch_v1_practice.db` | 当前有效 |
| 同步出站队列 | `SyncDatabase` | `memorize_words_arch_v1_sync_outbox.db` | 当前有效 |
| 悬浮背词展示记录 | `FloatingDatabase` | `memorize_words_arch_v1_floating.db` | 当前有效 |
| 旧词库实现 | `WordDatabase` | `memorize_words_arch_v1_word.db` | 仓库保留，当前未编译 |

证据：

- `core-database/src/main/java/com/chen/memorizewords/core/database/DatabaseContracts.kt`
- `core-database/src/main/java/com/chen/memorizewords/core/database/DestructiveRoomDatabaseFactory.kt`
- `data/build.gradle.kts`
- `settings.gradle.kts`

### 1.3 全局迁移策略

所有通过 `DestructiveRoomDatabaseFactory` 创建的数据库都会调用：

```kotlin
fallbackToDestructiveMigration(dropAllTables = true)
```

这意味着 Room 版本升级但缺少显式迁移时会删除并重建表。当前所有 `@Database` 均设置 `exportSchema = false`，仓库内未发现显式 `Migration`/`AutoMigration` 配置。

这是本设计中最高优先级的数据安全风险，详见第 5 节。

## 5. 设计缺陷审查清单

### P0. 破坏性迁移会导致本地数据丢失

- 证据：`DestructiveRoomDatabaseFactory` 对所有 Room 数据库调用 `fallbackToDestructiveMigration(dropAllTables = true)`。
- 影响：升级数据库版本且没有显式 Migration 时，Room 会删除所有表；学习进度、收藏、练习记录、同步队列等本地数据可能丢失。
- 建议：为每个数据库建立显式迁移策略；至少对用户数据表禁用破坏性迁移；区分可重建缓存和不可丢失用户数据。

### P0. `PracticeDatabase` 已经是 version 2，但无迁移和 schema 记录

- 证据：`PracticeDatabase` version = 2，`exportSchema = false`，未发现 `Migration`。
- 影响：已经发生过 schema 演进，但没有可审计历史。线上用户从旧版本升级时更容易触发破坏性重建。
- 建议：恢复 Room schema export；补齐 v1 -> v2 的迁移或确认历史版本未发布。

### P1. 所有数据库 `exportSchema = false`

- 证据：所有 `@Database` 均设置 `exportSchema = false`。
- 影响：无法用 Room schema JSON 做结构 diff、迁移测试和审计；后续改表容易漏迁移。
- 建议：开启 `exportSchema = true`，配置 schemaLocation，并将 schema 纳入版本控制。

### P1. 词库模型重复：`data-wordbook` 与 `data-word`

- 证据：`data-wordbook` 与 `data-word` 均定义 `words/word_definitions/word_examples/word_forms/word_roots/...` 等同名同义表。
- 影响：如果后续误启用 `data-word`，会出现两套词库数据源、两套数据库文件和重复 repository，造成数据不一致或依赖混乱。
- 建议：若 `data-word` 已废弃，应删除或归档；若计划保留，应抽取共享 schema 或明确迁移路线。

### P1. 学习进度表在多个数据库中重复

- 证据：`WordBookDatabase` 和 `StudyDatabase` 都定义 `word_learning_state`、`word_book_progress`。
- 影响：同一业务概念可能出现两个权威来源。跨库同步或镜像失败时，UI、同步、练习推荐可能读到不同状态。
- 建议：明确单一写入源；另一个库若是历史兼容，应加废弃说明；若是镜像，应记录同步触发点、冲突策略和一致性保障。

### P1. MMKV 使用 default 多进程实例承载多业务域

- 证据：`data-sync` 提供 `MMKV.defaultMMKV(MMKV.MULTI_PROCESS_MODE, null)`，账号、学习、同步、练习、悬浮设置均写入该实例。
- 影响：key 命名冲突、清理范围误伤、用户态和设备态边界不清晰。
- 建议：按业务域或作用域拆分 MMKV 实例；为 key 建立命名规范和集中清单；区分用户数据、设备配置、缓存数据。

### P1. 启动清理逻辑调用 `mmkv.clearAll()`

- 证据：`LocalAssetResetter` 会执行 `mmkv.clearAll()`，再恢复认证快照。
- 影响：如果新增 MMKV key 未纳入快照/恢复策略，可能在 reset 版本升级时被静默删除。
- 建议：将 reset 策略显式列出白名单/黑名单；为新增 key 要求声明作用域和 reset 行为。

### P2. 部分表缺少外键，依赖应用层维护一致性

- 例子：`StudyDatabase.word_learning_state`、`StudyDatabase.word_book_progress` 无 FK；`PracticeDatabase.exam_practice_item.word_id` 未关联词库表；`PracticeReportEntity.sessionId` 未关联 `practice_session_record.id`。
- 影响：删除词书或单词后可能留下孤儿记录；跨物理数据库无法建立 FK，只能靠应用层清理。
- 建议：跨库引用应在文档中标注“逻辑外键”；删除/重置流程需要覆盖孤儿数据清理测试。

### P2. 内容数据和用户数据混库

- 证据：`WordBookDatabase` 同时包含词书内容表、词库内容表、用户学习状态和用户扩展信息。
- 影响：词书内容更新、下载失败、清库重建可能影响用户数据；迁移粒度较粗。
- 建议：区分可重建内容缓存与不可丢用户状态；若继续混库，应对用户数据表提供独立备份/同步保障。

### P2. 日期字段语义需要统一

- 证据：多表以 `date` 作为主键或查询字段：`check_in_record`、`daily_study_duration`、`daily_practice_duration`、`floating_word_display_record`、`word_study_records`。
- 影响：如果日期格式、时区、业务日边界不统一，会影响连续打卡、每日统计和悬浮记录。
- 建议：文档化 `date` 格式、时区来源和业务日边界；与 `CheckInConfigDataSourceImpl` 的 timezone/day boundary 策略对齐。

## 6. 事务与一致性

已发现的事务使用点：

- `WordBookDatabase`：`RoomWordBookTransactionRunner`、`WordBookRepositoryImpl`、`WordBookPagePersister`、`WordBookContentDownloader`、`CurrentWordBookUpdateWorker`。
- `StudyDatabase`：学习状态、学习记录、收藏、快照重置等 repository 使用 `withTransaction`。
- `PracticeDatabase`：练习时长、会话记录、考试题缓存、快照重置使用 `withTransaction`。
- `SyncDatabase`：`SyncOutboxStore` 使用 `withTransaction` 处理队列状态。
- `FloatingDatabase`：展示记录保存和快照重置使用 `withTransaction`。

限制：

- Room transaction 只能覆盖单个物理数据库。
- `WordBookDatabase`、`StudyDatabase`、`PracticeDatabase`、`SyncDatabase`、`FloatingDatabase` 之间没有跨库事务。
- 跨库一致性需要由应用层流程、同步队列或重置流程保证。
